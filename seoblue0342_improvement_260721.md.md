## 3. 앱 A — G-Drive

Google Drive API는 OCI Ubuntu 서버에 이미 구성되어 있다고 가정하되, 연결 주소나 인증 정보를 임의로 추측하지 않는다. 기존 프로젝트, 배포 설정, 환경변수 예제 및 서버 구성을 조사해 실제 연결 방식을 확인한다. 비밀키, 토큰, 비밀번호는 소스에 하드코딩하거나 브라우저로 노출하지 않는다.

Stella DB 앱을 참조해 기존 UI/UX와 데이터 처리 방식을 최대한 활용한다. 최소 기능은 다음과 같다.

- 로그인한 사용자만 접근 가능
- Google Drive 폴더 및 파일 목록 조회
- 폴더 이동과 상위 경로 탐색
- 파일명, 유형, 수정일, 크기 표시
- 파일 검색
- 파일 다운로드
- 가능한 경우 업로드, 새 폴더 생성, 이름 변경, 이동, 삭제 기능
- 삭제처럼 데이터 손실 가능성이 있는 작업은 명확한 확인 절차 적용
- 로딩, 빈 결과, 권한 부족, API 오류 상태를 사용자에게 이해하기 쉽게 표시
- 모바일과 데스크톱에서 사용할 수 있는 반응형 화면

쓰기 기능은 Google Drive API 권한 범위가 허용하는지 먼저 확인한다. 권한이나 API가 준비되지 않은 기능은 가짜로 구현하지 말고, 읽기 전용으로 안전하게 동작시킨 뒤 제한 사항과 필요한 후속 설정을 문서화한다.

## 4. 앱 B — Obsidian Download

사용자가 ChatGPT 또는 Claude의 대화 링크/공유 링크를 입력하면, 접근 가능한 내용을 가져와 Obsidian에서 읽을 수 있는 Markdown 파일로 변환하고 다운로드할 수 있게 한다.

최소 기능은 다음과 같다.

- ChatGPT 및 Claude 링크 입력
- URL 형식과 지원 도메인 검증
- 대화 제목, 출처 URL, 내려받은 시각 등 기본 메타데이터를 YAML frontmatter에 기록
- 사용자와 AI 메시지의 역할이 구분되는 읽기 좋은 Markdown 생성
- 코드 블록, 표, 목록, 링크가 가능한 범위에서 보존되도록 변환
- 파일명에 사용할 수 없는 Windows 문자를 안전하게 처리
- 미리보기 후 다운로드
- 기본 저장 대상은 `C:\obsidian\download`
- 사용자가 다른 저장 경로를 선택하고 이후 기본값으로 저장 가능
- 중복 파일명이 생길 경우 덮어쓰지 않고 안전한 이름 생성
- 비공개 대화, 로그인 필요 페이지, 만료 링크, 접근 거부 시 명확한 안내
- 사용자의 대화 내용이나 링크를 불필요하게 서버 DB나 로그에 보관하지 않음

### 브라우저 저장 경로 제한에 대한 필수 처리

일반 웹 브라우저는 보안상 웹사이트가 사용자의 `C:\obsidian\download` 같은 임의의 로컬 폴더에 묻지 않고 자동 저장하는 것을 허용하지 않을 수 있다. 이를 무시하거나 불가능한 동작을 구현했다고 주장하지 않는다.

먼저 현재 저장소에 로컬 에이전트, 데스크톱 앱, 브라우저 확장, File System Access API 등 기존 수단이 있는지 조사한다. 그 후 다음 중 현재 환경에 가장 안전하고 유지보수하기 쉬운 방식을 선택한다.

1. Chromium 계열 브라우저의 File System Access API를 사용해 최초 1회 폴더를 사용자가 선택하고 권한을 부여한 뒤 저장
2. Windows 로컬 companion agent가 필요한 경우 최소 권한으로 별도 구성하고, localhost 인증·허용 도메인·경로 검증을 적용
3. 위 방식이 불가능하면 일반 다운로드 기능을 제공하고 브라우저 다운로드 기본 경로를 `C:\obsidian\download`로 설정하는 방법을 안내

경로 변경 UI와 저장 방식의 보안 경계를 명확히 설명한다. 서버가 임의의 Windows 경로에 직접 기록하는 식으로 구현하지 않는다.

ChatGPT/Claude 페이지를 가져오는 방식도 공식 공유 페이지 또는 사용자가 제공한 접근 가능한 콘텐츠만 사용한다. 인증 쿠키 탈취, 로그인 우회, 비공개 콘텐츠 무단 접근은 구현하지 않는다. 외부 페이지 구조 변경에 대비해 파서와 오류 처리를 분리하고 테스트한다.

## 5. 인증 및 계정 보안

`C:\codex\stella-clover`의 계정 관련 구현을 조사하고, 안전하며 현재 저장소와 호환되는 부분을 참조한다.

- 아이디 `yesblue0342`, 초기 비밀번호 `admin`

보안상 다음 조건을 반드시 적용한다.

- 평문 비밀번호를 Git 저장소, 프런트엔드 번들 또는 일반 로그에 저장하지 않는다.
- 초기 계정은 안전한 seed/migration 또는 배포 시 환경변수 방식으로 생성하고 비밀번호는 강한 해시로 저장한다.
- 최초 로그인 직후 비밀번호 변경을 강제한다.
- 초기 비밀번호를 변경하기 전에는 핵심 기능 및 관리자 기능 사용을 제한한다.
- 세션 쿠키는 운영 환경에서 `HttpOnly`, `Secure`, 적절한 `SameSite`를 적용한다.
- 로그인 시도 제한, 세션 만료, 로그아웃, CSRF/XSS/SQL injection 방어를 점검한다.
- 인증되지 않은 사용자가 G-Drive API, 파일 목록, 다운로드 API에 직접 접근하지 못하도록 서버 측에서 검증한다.

초기 비밀번호는 설치 직후 임시로만 존재해야 한다. 운영 배포 전 변경 필요성을 README와 배포 체크리스트에 눈에 띄게 기록한다.
